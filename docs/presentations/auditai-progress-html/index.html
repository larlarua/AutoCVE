ReAct State Machine

ReAct 循环更像“状态机调度”

源码里不仅有“思考-行动-观察”循环，还有继续、压缩恢复、收敛、停止等多类状态转移。

1. 构造上下文 拼接系统提示、技能路由、记忆注入与当前会话历史
2. 模型决策 产出回答、tool_use、或进入压缩 / 恢复分支
3. 工具执行 经权限判断、guardrails 与并发编排后执行工具
4. 观察整合 把工具结果、hook 产物、handoff 与记忆写回 transcript
5. 收敛输出 进入 report_finalization / final-only mode，停止新工具，返回 JSON 结论

常见继续条件

  • 模型明确返回新的 tool_use，转入下一轮
  • Finding 还有 runnable candidates 或 follow-up budget
  • token budget 允许继续，系统发出 continuation nudge
  • 命中 max_output_tokens，进入续写恢复
  • prompt 过长，触发 reactive compact / collapse drain 后重试

常见终止条件

  • 本轮不再调用 tool_use，且没有 transition，直接 completed
  • 进入 report_finalization 或 final-only mode
  • 达到 max_turns 上限（可配置）
  • 命中 stop hook、blocking limit、prompt-too-long 等硬停止
  • 同类无进展工具模式重复 3 次，或同一工具连续报错 3 轮
核心价值：AuditAI 的 ReAct 不是“多转几轮”，而是“按状态与条件收敛” 03 / 10

统一注册

运行时工具先收口到统一 Registry，模型看到的是标准化接口，而不是散落脚本。

Context Compaction

当前已有的上下文压缩机制

目标不是单纯“省 token”，而是让长时审计跑得更久，同时尽量不断掉关键证据。

1. tool_result_budget 先截断冗长工具输出，把无关细节留在边缘
2. history_snip 超过阈值后，只保留最近若干条消息，其余变成边界提示
3. microcompact 对超长 tool result 就地微压缩，保留“看得懂”的前段信息
4. context_collapse 把较早轮次折叠成 summary marker，关键信息改成可回忆摘要
5. auto_compact 逼近上下文阈值时自动触发总结，把旧对话压成 compact summary
6. reactive_compact / collapse_drain 真的溢出时，再走补救式压缩与重试；最后还有 blocking limit 兜底

平时轻量化

平时先用裁剪、snip、microcompact 这类低成本动作，不轻易动全量总结。

逼近阈值再总结

当 token 使用逼近 auto-compact threshold，系统才做真正的 conversation summary。

异常场景可恢复

prompt too long、媒体过大、输出截断时，会优先恢复再继续，而不是直接失败。

实际意义：AuditAI 正在从“短对话助手”升级为“能跑长流程”的审计执行器 06 / 10

Positioning

与传统源码审计工具的差异与互补

AuditAI 不是要替代全部扫描器，而是把“规则扫描、深度推理、人工协同、最终验证”串成一条闭环。

传统工具更擅长

  • 批量跑规则，发现广泛可疑点
  • 稳定覆盖常见弱点与依赖风险
  • 在超大项目里快速给出第一轮扫描面

AuditAI 更擅长

  • 跨文件理解业务逻辑与权限边界
  • 把 source 到 sink 利用链追得更完整
  • 在同一会话中接入人工追问与验证

最优组合方式

  • 扫描器负责“广”，AuditAI 负责“深”
  • 先工具铺面，再用 Finding/Verification 收敛
  • 最后把结果沉淀成报告、记忆和 Skill 资产
一句话：传统工具给线索，AuditAI 负责把线索推进成可汇报、可验证、可复盘的结论 08 / 10

Current Gaps

目前的缺陷

从源码实现来看，AuditAI 已经有了比较完整的骨架，但还处在“从可用走向好用、稳用”的阶段。

稳定性与成本 Finding Runtime、压缩恢复、工具编排链路已经很完整，但长流程成本和边界稳定性还需要继续打磨。
动态验证门槛 Verification 能做 harness 与 sandbox 验证，但对项目运行环境、依赖、样本准备仍有要求。
资产沉淀仍在扩库 Skill、规则、模板都已接入运行时，但高质量行业模板与经验资产还需要继续扩充。
平台处于双栈并进期 当前既有固定阶段多 Agent，也有 Finding Runtime 新栈，后续还需要继续收敛统一体验。
判断：这些缺陷更像阶段性工程问题，不是方向性问题 09 / 10
