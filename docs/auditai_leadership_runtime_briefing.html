ReAct Demo

ReAct 循环示范的动画实现：从线索到漏洞闭环

用一个领导最容易理解的案例来讲：上传接口是否存在路径穿越 / 任意文件覆盖。重点不是“瞬间判断”，而是 Agent 如何一轮轮收证、补证、闭合利用链。

Thought

怀疑上传接口可能存在路径穿越或任意文件覆盖

先基于接口语义建立假设，而不是立即下结论。

Action

调用 Grep 搜关键函数，再用 Read 读控制器与文件处理逻辑

快速锁定实现位置、入参与写文件路径拼接点。

Observation

发现用户输入文件名参与路径拼接，缺少规范化校验

获得“像漏洞”的代码证据，但还没有闭环成真实利用。

Thought

继续确认是否真能写到敏感路径，以及调用链是否可达

从单点 sink 继续向外扩，看路由、鉴权与边界是否成立。

Action + Observation

继续 Read / Grep 上游路由、鉴权逻辑，必要时用 PowerShell/Bash 做轻量验证

确认外部可达、权限边界不足或鉴权可绕过，形成利用链。

Final Answer

输出高置信 finding：成因、影响、利用条件、证据位置、修复建议

最终交付的是可验证、可复核、可转报告的结果，而非泛泛告警。

不是一次性判断，而是“逐轮收证”

Runner 每轮都能根据新 observation 调整搜索方向，避免过早下结论。

不是只看 sink，而是“闭合利用链”

从入口、鉴权、边界、影响面一路往后追，直到证明“为什么真能打”。

不是泛泛告警，而是“高置信输出”

目标是把 finding 打磨成接近研究员结论，而不是堆出更多怀疑点。

Harness Capabilities

Harness 配件：工具、Skill、上下文压缩

T1

工具与工具编排

统一挂在 Runtime Tool Registry 下，由 Tool Orchestrator 做参数校验、权限检查、执行记录与并发调度。

Read Glob Grep Write Bash PowerShell Skill TodoWrite AskUser ToolSearch Guardrail 并发调度
S2

Skill 机制与渐进式披露

Skill 不是一次性全量塞进上下文，而是先给导航，再按阶段展开正文、参考资料与脚本。

  • catalog → body → references → examples → scripts → full
  • 前期先帮助模型“找到方向”，后期再注入细节。
  • 避免上下文被技能文档淹没，提高有效 token 密度。
C3

上下文压缩与适用场景

Runtime 内置多层压缩策略，目标是在长链调查中尽量保留关键证据，不让 Agent 因 token 压力失焦。

  • tool result budget：大文件、多命中搜索先裁剪。
  • history snip / microcompact：保留最近关键轮次。
  • context collapse：把老上下文折叠为摘要。
  • auto / reactive compact：prompt too long 时自动压缩重试。
  • post-compact reinjection：补回关键附件、skill 片段与 plan 状态。
Why AuditAI

为什么 AuditAI 不只是“更聪明的扫描器”

传统扫描器回答“哪里像漏洞”，AuditAI 更努力回答“这里为什么真能打成漏洞”。

维度 传统扫描工具 AuditAI
分析方式 标准化规则覆盖，偏模式匹配。

推理 + 工具 + 多轮求证，更像研究员逐步排查。

关注对象 容易停留在单点危险函数。

更关注从入口到影响面的完整利用链。

优势场景 规则成熟、结构稳定、覆盖可量化的场景。

复杂逻辑漏洞、权限边界、鉴权绕过、多跳调用链。

产出目标 最大化告警数量。

尽快收敛到高置信漏洞，而不是堆疑似点。

工程价值 适合作为基础覆盖层。

已具备快速挖掘开源项目可提交 CVE 线索的工程化能力。

主结论

强在“推理 + 工具 + 利用链闭环”

对复杂上下文理解、跨文件链路、业务逻辑漏洞，AuditAI 的有效性明显高于单纯规则扫描。

跨文件推理 业务逻辑理解 高置信 finding
“传统扫描工具回答哪里像漏洞，AuditAI 更努力回答这里为什么真能打成漏洞。”
Current Limits

当前局限：覆盖率不可控，偏向高质量但不保证全量发现

主叙事

更像“高水平研究员助手”，还不是“确定性覆盖引擎”

AuditAI 已经具备打点突破能力，适合找到高价值、可打穿的真实漏洞；但它仍然受到时间预算、上下文预算、工具稳定性与技能适配度的共同影响，暂时无法承诺稳定、可量化的全覆盖。

高价值漏洞优先 覆盖边界尚不稳定
一句适合领导的表述：它已经具备“打点突破”的能力，但离“可度量、可证明覆盖”的平台化扫描能力还有差距。
漏洞覆盖率不可控

Agent 会受时间预算、上下文预算、搜索路径选择影响，无法像规则扫描那样给出清晰覆盖边界。

攻防差异性明显

更适合“找出一个能打穿的真实漏洞”，不适合承诺“所有同类漏洞都已扫到”。

收敛质量依赖上下文与 Skill

项目结构复杂、技能库不贴合、压缩过早，都可能让深层线索丢失。

工程稳定性仍在增强

长链调查中的预算控制、错误恢复、策略切换仍在持续打磨。

Roadmap

后续规划：从“能挖到”走向“更稳定、更可控、更可规模化”

主线 01

覆盖率工程化

建设标准漏洞基准集与项目级评测集，引入任务完成率、漏洞族覆盖率、有效利用链闭合率等指标。

主线 02

Runtime 强化

继续完善状态机、错误恢复、策略切换、长上下文调度，让 Agent 少走弯路、少空转。

主线 03

Skill 与 Harness 扩展

补齐更多漏洞族技能、项目类型模板、工具编排策略，让 Agent 更快进入有效调查路径。

主线 04

报告与闭环平台化

把 finding、report bundle、漏洞管理同步、复核反馈串成完整流水线，形成可复盘资产。

Short Term
提高单项目高价值漏洞产出效率

优先提升“找到真漏洞”的速度与稳定性，形成更强的单点突破能力。

Mid Term
提升覆盖率可解释性和稳定性

让结果不仅更好，还更可量化、可复盘、可被管理层理解。

Long Term
形成 AI 驱动的漏洞挖掘与管理平台

把挖掘、验证、报告、管理、反馈统一到平台级闭环中。
